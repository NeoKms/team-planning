# Decisions

## 2026-06-11: локальная память проекта

Решено хранить устойчивый AI-контекст в `tools/ai/memory`, потому что служебная директория `.agents` недоступна для записи в текущей среде.

Правила для агента вынесены в `AGENTS.md`.

## 2026-06-11: базовые продуктовые договоренности

- Приложение развивается как client-only инструмент без backend: хранение и вычисления выполняются на клиенте.
- Для расчетов и ввода длительности используется минута как базовая единица времени; в интерфейсе можно форматировать в дни, часы и минуты.
- Один рабочий день по умолчанию равен 6 часам.
- Спринт при планировании выбирается на 1 или 2 недели.
- Пятница всегда считается нерелизным днем.
- План реализации является живым: его нужно корректировать по мере динамичного добавления фич и уточнений.

## 2026-06-11: Tailwind CSS для дизайна

- Для UI используется Tailwind CSS через официальный Vite-плагин `@tailwindcss/vite`.
- Новые экраны и компоненты нужно верстать utility-классами Tailwind, а scoped CSS оставлять только для случаев, где utility-классов недостаточно.
- Глобальный CSS entrypoint: `src/assets/main.css`.

## 2026-06-12: автоназначение только в результате расчета

- Если у этапа нет исполнителя, scheduling engine может автоназначить подходящего участника по роли, чтобы расчет не падал раньше времени.
- Автоназначение записывается только в `AllocationResult.stages` и сопровождается `info` warning; исходный `WorkItem.assignments` не мутируется.
- Ручные назначения остаются источником истины для формы планирования. Рекомендации по переназначению показываются как warnings, а не применяются автоматически.

## 2026-06-12: design review выполняет внешний дизайнер

- В задачах не выбирается конкретный reviewer для design review.
- Если design review нужен, расчет планирует его на обезличенного внешнего исполнителя `Дизайнер`.
- `Дизайнер` может появляться в Gantt как отдельная строка, но не является участником команды и не участвует в рекомендациях по перераспределению инженерной загрузки.
- Design review блокирует только release support этой задачи или epic-release, если задача входит в epic; QA testing и зависимости следующих задач от него не ждут.

## 2026-06-12: модель и поведение отпусков

- Отпуска хранятся как `VacationPeriod[]` в `PlanningState` (верхний уровень, не вложено в участника): поля `id`, `memberId`, `startDate`, `endDate` (ISO строки).
- Загрузка старых сохранённых данных без поля `vacations` мигрируется автоматически: добавляется пустой массив через `migrateState()` в `loadState()`.
- Scheduling engine: дни отпуска дают 0 ёмкость сотрудника; при полном отпуске на весь спринт сотрудник исключается из автоназначения (фильтр по `sprintCapacityByMemberId`). Ручные назначения не блокируются — scheduler просто переносит работу на незанятые отпуском дни и при необходимости уходит в overflow.
- `SprintPlanningView` не допускает выбора ответственным участника, у которого отпуск охватывает все рабочие дни спринта (`membersOnFullSprintVacation`).
- `useAllocationReport` исключает таких участников из строк Gantt; остальные участники отображаются с серыми барами/фоном на отпускные дни.
- Изменение любого отпуска очищает `allocationResults` для всех спринтов команды участника.

## 2026-06-12: форматирование дат по локали браузера

- Все отображения ISO-дат (YYYY-MM-DD) в UI проходят через `formatDate(isoDate)` из `src/domain/planning.ts`.
- Реализация: `new Date(year, month-1, day).toLocaleDateString()` — используется локаль браузера, формат не хардкодится.
- Не использовать ручное `DD.MM.YYYY` форматирование; при добавлении новых дат в шаблоны — только через `formatDate`.

## 2026-06-14: качество scheduling — critical path и fan-out

- Scheduling engine теперь вычисляет `criticalPath(stage) = estimateMinutes + max(criticalPath(downstream))` через DFS по обратному графу зависимостей.
- Этапы с длинным критическим путём и высоким fan-out (кол-во параллельно разблокированных этапов) получают дополнительный score и планируются раньше.
- Контекстные бонусы `SAME_WORK_ITEM_SCORE_BONUS`, `SAME_ASSIGNEE_SCORE_BONUS`, `CONTEXT_SWITCH_SCORE_PENALTY` убраны: они нарушали свойство "equal cumulative score for any permutation", из-за чего beam search выбирал "красивую" последовательность вместо оптимальной по makespan.
- Доказательство: без контекстных бонусов суммарный score любого порядка расстановки N этапов одинаков; serial тайбрейкер стабильно выбирает вариант, где лучшие единицы идут первыми.

## 2026-06-14: оптимизация через objective и формат рекомендаций

- Beam search остаётся генератором допустимых кандидатов порядка scheduling units, но финальный выбор порядка делается по результату лёгкой симуляции allocation и objective-score, а не только по предварительному heuristic score.
- Веса, размеры beam и пороги рекомендаций живут в `SchedulingPolicy`; текущий публичный дефолт — `balanced`.
- Автоназначение пустых исполнителей выбирает кандидата по earliest completion на текущем календаре. Это не мутирует `WorkItem.assignments`, только `AllocationResult.stages`.
- Ручные назначения считаются authoritative. Scheduler может предложить замену исполнителя только как advisory `assignmentRecommendations`; применение замены должно быть явным пользовательским действием, если будет реализовано позже.
- Не показывать общий блок объяснимости вида «Почему этапы стоят так», пока он не даёт конкретного tradeoff/action. Низкосигнальные объяснения зависимостей и release-ожиданий создают шум в отчёте. Для доверия к расчёту использовать warnings, measurable assignment recommendations и summary-метрики.

## 2026-06-14: импорт без случайного затирания

- Импорт отдельной команды, спринта или результата расчёта считается merge-операцией и не должен удалять/перезаписывать уже существующие данные пользователя.
- При совпадении ID или названий импортируемые team/sprint добавляются как копии с новыми ID и суффиксом `(импорт)`; все внутренние ссылки snapshot должны быть перепривязаны на новые ID.
- Полный импорт рабочей области (`full` envelope или legacy raw `PlanningState`) остаётся destructive replace. Если в текущей рабочей области есть данные, перед применением нужно показать предупреждение и предложить резервную копию.

## 2026-06-21: QA test cases и QA testing выполняет один человек

- В рамках одной задачи исполнитель `qa-test-case-writing` и `qa-testing` должен совпадать.
- Модель формы остается с одним полем `assignments.qa`, поэтому ручное назначение автоматически применяется к обоим QA-этапам.
- Если QA не выбран вручную, scheduler сначала автоназначает исполнителя на test case writing, а затем наследует его для QA testing вместо независимого выбора более свободного QA.


- `scheduleSprint` остается чистой публичной функцией, но внутренний порядок планирования выбирается scored beam-search по готовым scheduling units.
- Release support stages одной задачи или epic объединяются в единый release unit: старт синхронный, длительность сопровождения остается индивидуальной по ролям.
- Scoring должен явно учитывать приоритет, release/high-priority urgency, раннее размещение релизов, минимизацию лишних переключений контекста и продолжение того же work item/assignee.
- Минимум 80% загрузки остается warning-метрикой качества плана: optimizer не добавляет фиктивную работу, а сообщает о недогрузе через существующие warnings.
