# Implementation plan

Этот план нужен как живая очередь работ. Его можно и нужно корректировать после новых решений, вайбкодинга и появления фич.

## Как Пользоваться Планом

- Перед началом работы прочитать `project-context.md`, `project-rules.md`, `decisions.md` и этот файл.
- Брать первый шаг со статусом `todo` или `in-progress`, если пользователь не просит конкретный шаг.
- После завершения шага обновлять статус, краткий результат и при необходимости следующий шаг.
- Не пытаться закрыть весь продукт одним заходом: каждый шаг должен оставлять приложение в рабочем состоянии.
- Если по ходу шага появляется более важная фича или меняется направление, обновить этот план, а не держать старый порядок как догму.

Статусы:

- `todo` — еще не начато.
- `in-progress` — начато, но не доведено до рабочего результата.
- `done` — реализовано и проверено на базовом уровне.
- `blocked` — нельзя продолжать без решения пользователя или внешней зависимости.

## Шаг 7. Первый Scheduling Engine

Статус: `done`

Цель: получить deterministic расчет нагрузки без сложной оптимизации.

Сделать:

- Преобразовать story/prod bug в этапы работ.
- Реализовать дефолтный workflow:
  1. анализ backend/frontend;

  3. frontend/ios/android после backend по умолчанию или параллельно при настройке;
  4. QA testing после разработки;
  5. релиз и сопровождение после тестирования.
- Учитывать зависимости между work items.
- Учитывать назначенных сотрудников и их дневную доступность.
- Раскладывать работу по рабочим дням спринта.
- Не ставить релиз на пятницу.
- Идти по приоритету work items.
- Возвращать структуру результата: слоты по сотрудникам и дням, warnings, overflow за пределы спринта.

Результат шага:

- Есть чистая функция или модуль scheduling engine, который можно тестировать отдельно.
- Для заданного спринта можно получить расчет нагрузки.
