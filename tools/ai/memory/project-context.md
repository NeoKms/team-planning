# Project context

`team-planning` — frontend-проект на Vue 3, Vite и TypeScript.

## Текущая структура

- `src/main.ts` — точка входа приложения.
- `src/assets/main.css` — глобальный CSS entrypoint с импортом Tailwind CSS и базовыми стилями.
- `src/App.vue` — корневой Vue-компонент с навигационным layout, `RouterView`, глобальным drag-and-drop импортом `.tpdata`/JSON-файлов, onboarding walkthrough и тостом запроса отзыва. Вычисляет `startsOnReportPath` синхронно во время setup для предотвращения флеша онбординга при открытии shareable-ссылки.
- `src/components/AllocationTimeline.vue` — переиспользуемый горизонтальный Gantt timeline для распределения и отчета.
- `src/components/AppConfirmDialog.vue` — глобальная Vue-модалка подтверждения действий; поддерживает обычный boolean confirm и выбор из нескольких действий через `useConfirmDialog().choose()`.
- `src/composables/useAllocationReport.ts` — общий слой подготовки данных распределения для экранов Allocation/Report: даты, группировка слотов, summary, форматирование и timeline-бары. Принимает опциональный параметр `stateOverride: Ref<PlanningState | null>` — при его наличии все данные берутся из него, а не из глобального стора (используется в `AllocationReportView` для изоляции данных из shareable-ссылки). Экспортирует `sprintTickets` (список tracker-задач для заведения в спринт с правилом нарезки: QA — 1 задача на всю оценку, остальные — max 1 рабочий день), `workItemOutcomes` (прогноз по задачам: релиз / полное тестирование / завершение разработки / в работе) и `assignmentRecommendations` для advisory-замен исполнителей.
- `src/composables/useConfirmDialog.ts` — общий composable для вызова подтверждений из любого компонента.
- `src/composables/useDataExchange.ts` — composable для импорта/экспорта данных. Включает: `getAllocationShareableLink` (генерирует URL `/reports/allocation/:sprintId#import=...`), `decodeAllocationState` (декодирует payload без записи в стор), `createShareableLinkFromState` (генерирует ссылку из произвольного `PlanningState`), подготовку импорта без записи в стор (`prepareImportFromText`/`prepareImportFromFile`) и применение подготовленного импорта (`applyImportDraft`).
- `src/composables/useReviewPrompt.ts` — composable для ненавязчивого промпта с отзывом: через 2 минуты нахождения на страницах результата распределения показывается тост «Как вам инструмент?» со ссылкой на Google-форму. Не чаще одного раза в 5 дней (ключ `team-planning:review-last-shown`). Работает в том числе при просмотре отчёта, открытого по shareable-ссылке.
- `src/domain/planning.ts` — доменные типы, константы планирования, `formatDate(isoDate)` для форматирования дат по локали браузера, `formatDuration(minutes)` для вывода длительностей в днях/часах/минутах и `priorityLabelByOrder(priority)` для маппинга числового приоритета в метку.
- `src/domain/scheduling.ts` — чистый deterministic scheduling engine: строит этапы работ, раскладывает слоты по сотрудникам/дням, учитывает отпуска участников, формирует warnings, overflow и advisory-рекомендации по замене исполнителей. Порядок работ строится через beam search scheduling units, после чего top-кандидаты сравниваются лёгкой simulation allocation по objective-score. Настройки весов/порогов вынесены в `SchedulingPolicy`; функция `getSprintWorkingDates` экспортирована для переиспользования в UI.
- `src/router/index.ts` — настройка Vue Router с маршрутами обзора, команд, отпусков, спринтов, рабочего места спринта, распределения, данных и отчета.
- `src/views/HomeView.vue` — рабочий обзор с быстрым доступом к основным разделам.
- `src/views/TeamsView.vue` — CRUD команд и участников: создание, inline-редактирование, удаление, роли, доступность и встречи по рабочим дням.
- `src/views/VacationsView.vue` — управление отпусками участников: добавление/редактирование/удаление периодов отпуска, отображение пересечений со спринтами (полный или частичный отпуск), фильтр по команде.
- `src/views/SprintsView.vue` — CRUD спринтов: создание, inline-редактирование, удаление, выбор команды и длительности 1/2 недели, переходы в план и распределение.
- `src/views/SprintPlanningView.vue` — рабочее место конкретного спринта: CRUD story/prod bug, epic-группировка, оценки, назначения, зависимости, порядок разработки, design review, релизное сопровождение и запуск распределения нагрузки. Участники на полный отпуск в рамках спринта не отображаются в списках назначений.
- `src/views/AllocationView.vue` — рабочий экран распределения нагрузки: запуск/перезапуск расчета, summary, загрузка сотрудников, табы «Timeline Gantt» / «Детализация по дням» (по умолчанию Gantt), серые бары отпусков, overflow и переход в отчет.
- `src/views/AllocationReportView.vue` — чистый report view по результату распределения для прямой ссылки `/reports/allocation/:sprintId`. При открытии через `#import=` хэш декодирует данные локально в `localState` (через `decodeAllocationState`) и передаёт их в `useAllocationReport` через `stateOverride` — глобальный стор и localStorage не затрагиваются. Кнопка «Скопировать ссылку на отчёт» расположена в заголовке отчёта рядом со summary-карточками. Показывает три таба: «Timeline Gantt», «Детализация по дням», «Рекомендации». Если есть advisory-рекомендации по исполнителям, таб «Рекомендации» подсвечивается зелёным actionable-акцентом и показывает отдельный бейдж `N замен`; warnings/info остаются отдельным бейджем.
- `src/views/DataExchangeView.vue` — отдельный раздел `/data` для полного импорта/экспорта, экспорта команд, спринтов и результатов распределения, загрузки демо-данных и очистки рабочей области. Перед destructive full import или импортом с совпадениями по названиям показывает предупреждение с выбором: скачать резервную копию и импортировать, импортировать без копии или отменить.
- `src/stores/planning.ts` — Pinia store для client-only planning state с localStorage persistence и actions для команд, участников, спринтов, work items, epics, отпусков и запуска расчета распределения. Демо-данные не содержит — они вынесены в `planning.seed.ts`.
- `src/stores/planning.seed.ts` — демо-данные для онбординга: функция `createSeedState()`, вспомогательные `nextMondayIso()` и `isoDateAddDays()`. Даты спринта и отпусков вычисляются динамически от текущей даты, чтобы демо всегда было актуальным.
- `public/` — статические файлы.
- `vite.config.ts` — конфигурация Vite, включая Tailwind CSS Vite plugin.
- `eslint.config.ts` — конфигурация ESLint.

## Заметки

- Базовый scaffold уже заменен на рабочее состояние: доменная модель с отпусками, planning store, seed-данные, основной layout, маршруты ключевых разделов, CRUD команд/участников, CRUD отпусков, CRUD спринтов, CRUD story/prod bug, epic-группировка внутри спринта, scheduling engine с design review/автоназначением/рекомендациями/учётом отпусков, экран Gantt-распределения с визуализацией отпусков, отдельный report view и раздел импорта/экспорта данных.
- Tailwind CSS подключен через `@tailwindcss/vite`; новый дизайн нужно делать на Tailwind utility-классах.
- Состояние хранится в `localStorage` под ключом `team-planning:v1`.
- Показ онбординга зависит от мажорной версии приложения: при изменении мажорной версии онбординг показывается снова. Используемый ключ `team-planning:onboarding-seen-version` хранит последнюю просмотренную мажорную версию. Версия приложения задаётся в `package.json` и инжектируется в код через `import.meta.env.APP_VERSION` (vite define в `vite.config.ts`).
- Shareable link на результат распределения генерируется через `getAllocationShareableLink` в `useDataExchange.ts` и ведёт на `/reports/allocation/:sprintId#import=...` — получатель попадает сразу в полный отчёт. Кнопка «Скопировать ссылку на отчёт» находится в заголовке `AllocationReportView`. Открытие по ссылке не изменяет рабочее пространство получателя: данные из хэша декодируются изолированно, стор и localStorage не затрагиваются.
- Импорт `sprint`, `allocation` и `team` должен быть non-destructive: при совпадении ID или имени импортируемые сущности добавляются как копии с новыми ID и суффиксом `(импорт)`, а зависимости внутри snapshot перепривязываются. Полный импорт (`full` и legacy raw `PlanningState`) остаётся заменой всей рабочей области и требует предупреждения/подтверждения, если у пользователя уже есть данные.
- Подтверждения действий должны идти через `useConfirmDialog`, а не через системные `window.confirm`/`alert`.
- При удалении команды удаляются связанные участники, спринты, задачи, epics, отпуска и результаты распределения; при удалении участника очищаются его назначения, слоты распределения и все его отпуска.
- При удалении спринта удаляются его задачи, epics и результаты распределения. При изменении команды, даты старта или длительности спринта результаты распределения очищаются; при смене команды назначения задач фильтруются по участникам новой команды.
- При создании, редактировании или удалении work item очищается результат распределения соответствующего спринта. При удалении work item очищаются зависимости других задач и привязки в epics.
- Work item может быть release-only вне epic: dev/QA оценки допускаются пустыми, если включено релизное сопровождение и есть положительные минуты сопровождения. Для задач внутри epic release support задается на уровне `Epic.releaseSupport`.
- Порядок work items внутри спринта хранится как уникальная последовательность `1..N`: при вставке/переносе задачи в позицию остальные задачи сдвигаются, при удалении порядок сжимается без дырок.
- При создании, редактировании или удалении epic очищается результат распределения соответствующего спринта. `Epic.workItemIds` и `WorkItem.epicId` синхронизируются в store; одна задача может быть только в одном epic.
- Для задач внутри epic release support должен учитываться один раз на уровне epic, через `Epic.releaseSupport`, а не суммироваться с каждой задачей.
- Scheduling engine запускается через `planningStore.calculateAllocationForSprint(sprintId)` и сохраняет `AllocationResult` со `stages`, `slots`, `warnings`, `overflow`, `policy` и `assignmentRecommendations`. После изменения work items, epics, отпусков или профилей участников (без смены команды/дат спринта) результат расчёта сохраняется, но спринт помечается как устаревший через `lastDataChangedAt`. `planningStore.isSprintAllocationStale(sprintId)` возвращает `true`, если данные изменились после последнего расчёта. Это отображается в UI: бейдж «Устарел» на кнопке «Открыть расчет» в `SprintPlanningView`, жёлтые баннеры-предупреждения в `AllocationView` и `AllocationReportView`.
- Engine учитывает положительные оценки dev/QA/design review/release, порядок приоритетов, зависимости work items, дневную доступность участника, отпуска участников, календарь встреч сверх процессного буфера 2 ч/день, рабочие дни спринта и запрет релиза в пятницу. Один scheduled slot ограничен максимумом 1 рабочего дня, то есть 360 минут: оценка 1080 минут раскладывается как три дневных слота, а невлезшие слоты попадают в overflow после текущего спринта. Отдельной оценки analysis в модели пока нет, поэтому analysis-этапы не создаются. Этап документации (`documentation`) является backend-этапом: зависит от `backend-development`, **не** блокирует QA testing, но блокирует release support (становится terminal-этапом), имеет наименьший scheduling-приоритет среди dev/QA стадий.
- Отпуска учитываются в scheduling engine: дни отпуска дают нулевую ёмкость сотрудника, работа автоматически переносится на незанятые отпуском рабочие дни. При полном отпуске на весь спринт сотрудник исключается из автоназначения. Engine добавляет `info` warning о частичном отпуске и `warning` о полном отпуске на весь спринт.
- `WorkItem` поддерживает optional design review: `requiresDesignReview` и `designReviewEstimateMinutes`. Старые persisted задачи без этих полей считаются валидными; scheduler и форма используют дефолты.
- Design review всегда считается внешней дизайнерской работой: в расчете он идет как обезличенный `Дизайнер`, не выбирается из команды и не грузит backend/frontend/QA участников.
- Design review блокирует только release support этой задачи или epic-release, если задача входит в epic. Он не блокирует QA testing и не становится зависимостью для следующих задач.
- При пустом исполнителе scheduler пытается автоназначить подходящего участника по роли только в рамках результата расчета и добавляет `info` warning. Автоназначение выбирает кандидата по earliest completion на текущем календаре и не мутирует исходный `WorkItem`.
- После расчета scheduler добавляет warnings о пересечениях QA testing/release support у нескольких сотрудников в один временной интервал. Ручные назначения не меняются автоматически: scheduler может вернуть advisory `assignmentRecommendations`, если пробная замена исполнителя той же роли даёт измеримую выгоду по objective-score.
- Не добавлять общий блок объяснений «Почему этапы стоят так», пока он не содержит конкретных tradeoff/action. Низкосигнальные объяснения зависимостей и release-ожиданий были признаны шумом; в отчёте остаются warnings и actionable-рекомендации.
- Scheduler добавляет подсказки по пустым QA/dev/release/design-review оценкам и предупреждает о role mismatch у вручную назначенного исполнителя.
- `AllocationView` и `AllocationReportView` получают `vacationDatesByMemberId` из `useAllocationReport` и: (1) в Timeline Gantt рисуют серый бар и серый фон колонки на дни отпуска; (2) в Gantt по дням показывают серый блок «🏖 Отпуск» в ячейке; (3) участники в полном отпуске исключаются из строк обоих Gantt.
- Форматирование дат в UI — через `formatDate(isoDate)` из `domain/planning.ts`, которая использует `toLocaleDateString()` браузера. Не хардкодить `DD.MM.YYYY` вручную.
- Форматирование длительностей в UI и рекомендациях — через `formatDuration(minutes)` из `domain/planning.ts`, чтобы минуты отображались как дни/часы/минуты при 6-часовом плановом дне.
- В план добавлены будущие требования: полноэкранный Gantt с timeline-барами, автоназначение исполнителей, рекомендации по переназначению, post-MVP ввод оценок в днях/часах, оценка документации для backend.
- При развитии проекта поддерживай этот файл в актуальном состоянии: добавляй важные модули, страницы, договоренности по структуре и команды.

## Продуктовая цель

Приложение должно помогать delivery manager планировать следующий спринт и видеть нагрузку людей по часам и дням недели с учетом ограничений: календарная занятость, пересечения с другими коллегами на тестах и релизах, зависимости тестирования от разработки, релиза от тестирования и дизайн-ревью, а также порядок выполнения задач внутри спринта.

Основной сценарий: пользователь создает или выбирает команду, создает спринт на 1 или 2 недели, добавляет story/prod bug, назначает исполнителей, задает оценки и зависимости, запускает распределение нагрузки, после чего видит диаграмму Ганта по каждому сотруднику и рекомендации по рискам планирования.

## Базовые продуктовые ограничения

- Приложение планируется без backend: вычисления и хранение должны выполняться на клиенте.
- Полный рабочий день сотрудника равен 8 часам, но в плановую доступность обычно закладывается 6 часов работы, потому что 2 часа считаются процессным буфером на встречи.
- Календарные встречи уменьшают плановую доступность только в части сверх 2 часов в конкретный день: если в календаре 90 минут встреч, берется исходная доступность; если 180 минут, из доступности вычитается 60 минут.
- Время для ввода и расчетов лучше хранить в минутах, а в интерфейсе выводить в днях, часах и минутах.
- Спринт создается на 1 или 2 недели, выбор делается при каждом планировании.
- Пятница всегда не релизный день.
- У delivery manager может быть несколько команд, поэтому нужны создание, список, редактирование и удаление команд.

## Команды и участники

В команде могут быть роли:

- frontend;
- backend;
- tech lead: считается backend, но имеет меньше доступных часов в день, чем обычный backend;
- qa;
- mobile ios;
- mobile android.

Нужно уметь добавлять и редактировать участников команды. Для разработчика должна быть отдельная настройка доступных часов в день и средней занятости встречами по дням недели, понедельник-пятница, в минутах. В поле доступности обычно ставится 6 часов, для tech lead можно меньше. Встречи учитываются в расчете как уменьшение effective capacity на соответствующую дату только в части сверх 2 часов.

Если в команде нет ios/android участников, соответствующие поля в story/prod bug можно скрывать или показывать заблокированными.

## Спринты

Нужен функционал:

- создать спринт;
- вывести список спринтов;
- удалить спринт вместе с его данными;
- назначить спринту одну из заранее созданных команд;
- при создании спринта добавить story/prod bug;
- после создания редактировать список story/prod bug.

## Story, Prod Bug И Epic

Основные сущности планирования:

- story;
- prod bug, иногда не привязан к story;
- epic, который может объединять несколько story и prod bug для релизов.

Для story/prod bug нужно задавать:

- приоритет, чтобы распределение и отображение списка шли по приоритету;
- верхнеуровневые оценки по backend, frontend, ios, android и qa;
- конкретных исполнителей из выбранной команды;
- зависимости от уже созданных story/prod bug: текущая задача должна выполняться строго после указанных;
- необходимость сопровождения релиза;
- время сопровождения релиза отдельно для ios, android, frontend, backend и qa.

Story/prod bug можно объединять в epic. Если story/prod bug входят в epic, время сопровождения релиза для каждого задействованного направления учитывается один раз на уровне epic, а не отдельно на каждой story/prod bug.

## Дефолтный workflow story/prod bug

По умолчанию этапы идут так:

1. Анализ: backend и frontend.
2. Backend-разработка и написание тест-кейсов QA могут стартовать параллельно.
3. Frontend, ios и android разработка.
4. Тестирование QA.
5. Релиз и сопровождение: frontend, backend, qa, ios, android.

Для QA всегда отдельно указывается:

- время на написание тест-кейсов;
- время на тестирование.

Написание тест-кейсов можно начинать параллельно с backend.

В story должна быть настройка, может ли backend/frontend/ios/android разработка идти параллельно. По умолчанию разработка последовательная и зависит от backend.

## Распределение Нагрузки

Нужна кнопка "распределить нагрузку", когда сотрудники уже назначены на story/prod bug. Распределение должно учитывать все назначенные на сотрудника story/prod bug и строить диаграмму Ганта по дням с детализацией каждого дня.

При расчете нужно учитывать:

- доступность сотрудников по рабочим дням;
- календарь встреч по дням недели, но только превышение сверх 2 ч/день;
- 6-часовой рабочий день;
- зависимости между story/prod bug;
- порядок этапов внутри story/prod bug;
- параллельность или последовательность разработки;
- зависимости QA от разработки;
- зависимости релиза от тестирования, дизайн-ревью и других ограничений;
- запрет релиза в пятницу;
- приоритет story/prod bug.

После распределения нужно показать:

- диаграмму Ганта по каждому сотруднику;
- рекомендации, если что-то не влезает в спринт или нарушает ограничения;
- отдельный отчет "кто чем занимается", пригодный для скриншота или отправки по прямой ссылке, чтобы участники могли завести себе задачи по распределению.

## Черновой План Реализации

План нужно считать живым и регулярно корректировать при появлении новых фич и уточнений.

1. Описать клиентскую доменную модель: команды, участники, спринты, story/prod bug, epic, оценки, этапы, зависимости, назначения и результаты распределения.
2. Настроить клиентское хранение данных, вероятно через Pinia и persisted/localStorage-слой.
3. Реализовать CRUD команд и участников.
4. Реализовать CRUD спринтов с выбором длительности и команды.
5. Реализовать CRUD story/prod bug внутри спринта, включая приоритеты, оценки, назначения, зависимости, релизное сопровождение и epic-группировку.
6. Сделать первый deterministic scheduling engine: разложить этапы по сотрудникам, рабочим дням и зависимостям без сложной оптимизации.
7. Добавить визуализацию распределения: Gantt-представление, дневная детализация, подсветка конфликтов и рекомендаций.
8. Добавить share/report view для отчета "кто чем занимается".
9. Постепенно усложнять ограничения: календарная занятость, пересечения с релизами/тестами других коллег, дизайн-ревью и более точные рекомендации.
