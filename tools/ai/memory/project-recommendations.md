# Рекомендации по развитию Team Planning

Дата анализа: 14 июня 2026
Последнее обновление: 14 июня 2026

## Краткое резюме

Проект уже находится в рабочем состоянии: есть доменная модель, Pinia-хранилище, CRUD основных сущностей, deterministic scheduling engine, расчет нагрузки, Gantt, отчеты и импорт/экспорт. Тесты планировщика и production build проходят.

Главная зона роста — не базовая работоспособность, а качество оптимизации расписания и мобильный UX. Планировщик уже использует сильную эвристическую базу: зависимости, приоритеты, critical path, fan-out, release urgency, отпуска, встречи и overflow. После обновления 14 июня 2026 beam search используется как генератор кандидатов, а финальный порядок выбирается по результату легкой simulation allocation и objective-score.

Визуально приложение выглядит цельно и достаточно аккуратно для рабочего инструмента, но на мобильных устройствах сложные поверхности остаются desktop-first: большая форма задачи, горизонтальный Gantt и верхняя навигация требуют отдельной адаптивной стратегии.

## Текущее состояние

### Что уже хорошо

- Scheduling engine отделен от UI и остается чистой функцией `scheduleSprint`.
- Расчет учитывает рабочие дни, пятницу как нерелизный день, дневную capacity, встречи сверх буфера, отпуска, зависимости задач, зависимости этапов, release support, design review и overflow.
- Есть beam search по scheduling units.
- Release support одной задачи или epic группируется и стартует синхронно по ролям.
- В алгоритме уже есть critical path и downstream fan-out scoring.
- Расчет сохраняется, но помечается устаревшим при изменении данных.
- UI показывает Gantt, загрузку сотрудников, риски, отчет и shareable-ссылку.
- Веса планировщика вынесены в `SchedulingPolicy`.
- Автоназначение выбирает кандидата по earliest completion на текущем календаре.
- Расчет может возвращать advisory-рекомендации по замене ручных исполнителей без автоматического изменения исходных назначений.
- Данные остаются client-only, что соответствует продуктовой цели.

### Основные ограничения

- Objective-функция уже добавлена, но пока живет внутри `scheduling.ts`; при росте сложности ее стоит выделить в отдельный тестируемый модуль.
- Рекомендации по замене исполнителей пока одиночные; пакетные замены и кнопка "Применить замену" еще не реализованы.
- Preset policy пока не выбирается в UI; используется дефолтный `balanced`.
- Мобильный Gantt является горизонтальной таблицей шириной до нескольких тысяч пикселей.
- Форма story/prod bug на телефоне превращается в длинную модалку с большим количеством секций.
- Верхняя навигация на mobile скроллится горизонтально, но это неочевидно пользователю.

## Рекомендации по оптимизации планирования

Статус на 14 июня 2026:

- пункты 1-4 выполнены в коде;
- пункт 5 не внедрен как отдельный блок UI: текущие объяснения признаны низкосигнальными;
- пункт 6 отложен по решению пользователя.

### 1. Добавить пост-оценку кандидатов расписания — выполнено

`getOptimizedStageOrder` оставляет beam search генератором кандидатов, но финально выбирает порядок по результату симуляции allocation.

Предлагаемый подход:

1. Beam search формирует top N кандидатов порядка scheduling units.
2. Для каждого кандидата запускается легкая симуляция allocation.
3. Каждый результат оценивается по objective-функции.
4. Побеждает кандидат с лучшей итоговой метрикой, а не с лучшим предварительным score.

Реализованные метрики objective-функции:

- минимум overflow minutes;
- минимум даты завершения release support;
- минимум общего makespan;
- минимум underload gap до 80%;
- минимум пересечений QA/release-sensitive работ;
- штраф за перенос high/critical задач за пределы спринта.

Пункт про "минимум простоев у критичных ролей" не реализован напрямую как отдельная метрика; частично покрывается makespan, release finish и underload gap.

### 2. Улучшить автоназначение через earliest completion — выполнено

Автоназначение выбирает кандидата по earliest completion на текущем календаре, с учетом уже занятых окон, отпусков, встреч и capacity.

Рекомендация:

- для незаполненного assignee собирать кандидатов по роли;
- для каждого кандидата делать пробное размещение stage в текущем календаре;
- выбирать того, кто завершит stage раньше;
- при равенстве выбирать меньшую итоговую загрузку или больший резерв capacity.

При равенстве учитываются текущая загрузка и резерв capacity.

### 3. Добавить рекомендации по замене заранее назначенных исполнителей — частично выполнено

Частый рабочий сценарий: часть задач уже заранее назначена на конкретных сотрудников, а часть задач остается без исполнителей. Текущий расчет уважает ручные назначения, но эти назначения могут быть неоптимальными с учетом отпусков, встреч, зависимостей, release support и уже занятых окон.

Важно не менять такие назначения автоматически. Ручное назначение может отражать контекст, которого нет в модели: знание домена, ownership, договоренности внутри команды, обучение нового сотрудника, риск передачи задачи. Поэтому лучше добавить отдельный блок рекомендаций: "если заменить исполнителя X на Y в таких-то этапах, расписание станет лучше на конкретную величину".

Предлагаемый подход:

1. Сначала построить базовый расчет с текущими ручными назначениями.
2. Найти stages, где есть ручной assignee и есть альтернативные кандидаты той же роли.
3. Для каждого stage или небольшой группы stages сделать пробную замену исполнителя.
4. Пересчитать affected schedule или выполнить быструю симуляцию.
5. Сравнить результат с базовым расчетом по objective-функции.
6. Показать только рекомендации с доказуемой и достаточно большой выгодой.

Примеры полезных рекомендаций:

- "Переназначить backend этап задачи A с Ивана на Олега: overflow уменьшится на 1 д, release завершится на день раньше".
- "Переназначить QA testing задачи B с Анны на Марину: загрузка Анны снизится со 115% до 82%, задача останется в спринте".
- "Перенести release support frontend с Петра на Светлану: релизная группа стартует во вторник вместо среды".

Метрики выгоды:

- уменьшение overflow minutes;
- более раннее завершение release support;
- снижение перегруза или underload gap;
- более раннее завершение high/critical задач;
- уменьшение пересечений QA/release-sensitive работ;
- более ровная загрузка участников одной роли.

Ограничители, чтобы рекомендации не были шумными:

- не предлагать замену, если выигрыш меньше заданного порога, например меньше 2 часов или меньше 5% objective-score;
- не предлагать замену между разными ролями, кроме явно разрешенных случаев вроде `tech-lead` как backend;
- не предлагать замену участника в полном отпуске или с нулевой capacity;
- не предлагать больше 3-5 рекомендаций за расчет;
- группировать несколько замен в один пакет только если вместе они дают лучший результат, чем по отдельности;
- помечать рекомендацию как advisory, а не как автоматическое изменение исходных `WorkItem.assignments`.

Реализовано:

- ручные назначения не меняются автоматически;
- scheduler делает пробные одиночные замены исполнителя внутри той же роли;
- сравнение идет по objective-функции;
- в `AllocationResult.assignmentRecommendations` попадают только рекомендации с достаточной выгодой;
- в report UI есть блок "Рекомендации по исполнителям";
- если такие рекомендации есть, таб "Рекомендации" подсвечивается отдельным actionable-акцентом и бейджем `N замен`.

Осталось:

- кнопка "Применить замену" как явное действие пользователя;
- пакетные рекомендации, если несколько замен вместе дают лучший результат, чем по отдельности.

### 4. Вынести веса планировщика в policy — выполнено

Веса вроде release bonus, critical path factor, fan-out factor, beam width и пороги рекомендаций оформлены как `SchedulingPolicy`.

Пример будущих preset:

- `release-first`: минимизировать риск позднего release;
- `balanced`: удерживать загрузку команды ближе к целевому диапазону;
- `critical-path`: быстрее закрывать цепочки зависимостей;
- `no-overflow`: агрессивно минимизировать работу после спринта.

Preset уже существуют в коде. UI для выбора стратегии пока не добавлен.

### 5. Добавить объяснимость результата — отложено / не делать в текущем виде

Сейчас пользователь видит warnings, но не всегда понимает, почему задача поставлена именно туда.

Можно добавить для каждого stage короткое объяснение:

- stage стартовал после зависимостей;
- выбран исполнитель, потому что у него раннее свободное окно;
- release перенесен из-за нерелизной пятницы;
- работа ушла в overflow из-за нехватки capacity;
- QA стартовал после dev-этапов.

Попытка добавить отдельный блок "Почему этапы стоят так" показала низкую ценность: сообщения в основном повторяли правила зависимостей и release-ожиданий, но не давали действия. Решение на сейчас:

- не сохранять `explanations` в `AllocationResult`;
- не показывать отдельный блок объяснений в отчете;
- доверие к расчету поддерживать через measurable `assignmentRecommendations`, warnings и summary-метрики;
- возвращаться к explainability только если сообщение объясняет конкретный tradeoff или предлагает действие.

### 6. Усилить тесты качества расписания — отложено

Текущие тесты покрывают важные инварианты, но можно добавить regression-наборы на качество:

- release work против bulk work одного приоритета;
- длинная цепочка зависимостей против короткой независимой задачи;
- partial vacation и full sprint vacation;
- meeting overload по конкретным дням;
- underload warnings;
- epic release support один раз на epic;
- role mismatch не ломает расчет;
- отсутствие QA estimate для dev-задачи дает info warning.

Отдельно стоит добавить тесты objective-функции для уже добавленной пост-оценки кандидатов.

## Рекомендации по визуальному интерфейсу

### 1. Упростить цветовую иерархию

Палитра в целом рабочая, но интерфейс местами перегружен бейджами, градиентами и цветными секциями.

Рекомендация:

- оставить яркий цвет только для primary action, статуса и критичных warning/error;
- карточки задач сделать более нейтральными;
- release/design/QA подсвечивать цветом, но не превращать каждую секцию в отдельный яркий блок;
- градиенты использовать реже, особенно в рабочих списках.

### 2. Заменить часть текстовых действий на иконки

В карточках много повторяющихся кнопок: редактировать, удалить, экспорт, пересчитать, открыть отчет.

Рекомендация:

- использовать иконки с tooltip для повторяемых действий;
- оставить текст только для primary action;
- в mobile action-группы сворачивать в menu.

Это снизит визуальный шум и поможет карточкам задач занимать меньше места.

### 3. Разделить ввод задачи на понятные шаги

Форма story/prod bug функционально полная, но на mobile слишком длинная.

Рекомендация:

- desktop: оставить grid, но визуально сгруппировать секции компактнее;
- mobile: использовать tabs или accordion;
- sticky footer оставить, но добавить текущий статус заполнения: "Оценки не заполнены", "Назначения опциональны", "Release включен";
- скрывать release поля до включения release support;
- скрывать design estimate до включения design review.

### 4. Улучшить пустые и проблемные состояния

Для delivery manager важно быстро понимать, что делать дальше.

Рекомендация:

- при пустом спринте показывать короткий next step;
- при underload показывать не только warning, но и возможные направления: "добавить задачи", "переназначить", "уменьшить scope";
- при overflow показывать список задач, которые ушли за спринт, и их причины.

## План мобильного адаптива

### Цель

На телефоне приложение не должно быть уменьшенной desktop-таблицей. Основной сценарий mobile: быстро посмотреть статус, загрузку, риски и детали по человеку/дню. Полное редактирование может быть поддержано, но должно быть пошаговым.

### 1. Глобальная навигация

Проблема: верхняя nav имеет горизонтальный scroll, последний пункт может уходить за viewport.

Рекомендации:

- добавить mobile bottom navigation для основных разделов;
- либо заменить nav на segmented tabs с видимым fade-индикатором scroll;
- summary-метрики в header свернуть в одну строку или скрыть за кнопкой;
- уменьшить высоту шапки на mobile.

Приоритет: высокий.

### 2. Рабочее место спринта

Проблема: action bar в заголовке переносится на несколько строк, карточки задач содержат много бейджей.

Рекомендации:

- оставить одну primary-кнопку "Новая задача";
- "Новый epic", "Экспорт", "Открыть расчет" убрать в menu;
- в карточке задачи показывать только priority, type, total estimate и title;
- назначения и planning details свернуть в disclosure;
- кнопки edit/delete заменить на compact icon actions.

Приоритет: высокий.

### 3. Форма story/prod bug

Проблема: modal занимает почти весь экран и содержит длинную форму.

Рекомендации:

- на mobile делать `AppModal` полноэкранным;
- header и footer формы сделать sticky;
- разбить форму на секции:
  - Основное;
  - Оценки;
  - Назначения;
  - Зависимости;
  - Design review;
  - Release support;
- использовать accordion или stepper;
- показывать compact summary вверху: total estimate, release on/off, blockers.

Приоритет: высокий.

### 4. Allocation summary

Проблема: экран расчета становится длинным, а ключевая информация размывается.

Рекомендации:

- наверху mobile-экрана показать 3-4 компактные метрики: in sprint, overflow, risks, underloaded;
- CTA "Открыть полный отчет" сделать компактнее;
- блок "Как считается расчет?" оставить свернутым;
- карточки сотрудников сделать горизонтально скроллящимися или компактным списком.

Приоритет: средний.

### 5. Mobile Gantt

Проблема: текущий Gantt имеет внутреннюю ширину больше 1000px, а для 2-недельного спринта может быть около 3000px. На телефоне это технически работает через scroll, но плохо как основной сценарий.

Рекомендации:

Вместо одной широкой таблицы добавить mobile view mode:

- "По дням": дата -> список участников -> задачи этого дня;
- "По людям": участник -> дни -> задачи;
- "Риски": только overflow, release, QA conflicts, underload.

Горизонтальный Gantt можно оставить как "табличный режим", но не делать его основным на mobile.

Приоритет: высокий.

### 6. Отчет

Рекомендации:

- mobile report должен открываться сразу в summary + risks;
- Gantt вынести во вкладку;
- tracker-задачи показывать группами по исполнителю;
- кнопку копирования ссылки держать в sticky header или compact action menu.

Приоритет: средний.

## Предлагаемый порядок внедрения
